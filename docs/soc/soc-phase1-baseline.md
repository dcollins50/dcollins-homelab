# **1. Purpose and Learning Objectives**

> **Status:** Live

---



This document serves two purposes. As a learning exercise, it walks through the reasoning behind each step so you understand not just what to do but why it matters. As a standard operating procedure, it defines the repeatable process for establishing a SIEM baseline that all future tuning work depends on.

## **1.1 Why This Phase Exists**

A SIEM that has just been deployed is not useful yet. It is producing alerts, but you do not know which of those alerts represent real threats, which are false positives, and which are noise generated by expected system behavior. Trying to respond to alerts before answering those questions is how analyst fatigue starts.

The baseline phase answers a fundamental question: what is normal? Until you know what normal looks like, you cannot recognize abnormal. This is the same reasoning that drives network traffic baselining, user behavior analytics, and endpoint detection tuning in production SOCs. The discipline is the same regardless of scale.

| **WHY** | In a real SOC, new SIEM deployments always go through a baseline period before alerts are acted upon. Skipping this step and treating every alert as actionable on day one is a reliable way to burn out your analysts and erode trust in the platform. |
| --- | --- |

# **2. Background: How Wazuh Alerts Work**

Before building anything, it helps to understand what you are looking at. Every Wazuh alert is a JSON document written to an Elasticsearch index named wazuh-alerts-4.x-YYYY.MM.DD. Each document contains a standard set of fields that you will use to filter, group, and sort data.

## **2.1 Key Alert Fields**

| **Field** | **What It Means** |
| --- | --- |
| rule.id | The unique identifier for the rule that generated this alert. This is your primary grouping key during baseline work. |
| rule.description | Human-readable description of what the rule detected. This is what you read to understand an alert. |
| rule.level | Severity from 0 to 15. Levels 0 to 6 are informational. Levels 7 to 11 are medium severity. Levels 12 to 15 are high severity and should always be investigated. |
| agent.name | The name of the host that generated the alert. Use this to identify which hosts are producing the most noise. |
| rule.groups | Categories the rule belongs to, such as authentication, fim, rootcheck, or docker. Use this to understand what type of activity is being detected. |
| @timestamp | When the event occurred on the agent. This is the field Kibana uses for time-based filtering. |

## **2.2 Severity Level Reference**

| **Levels** | **Category** | **Analyst Action** |
| --- | --- | --- |
| 0 to 3 | Informational | No action required. Used for logging and compliance. |
| 4 to 6 | Low | Review during baseline. Suppress or tune if generating consistent noise. |
| 7 to 11 | Medium | Investigate during triage. These are the primary tuning targets in Phase 2. |
| 12 to 15 | **HIGH** | Always investigate immediately. These should be rare in a healthy baseline. |

# **4. Procedure**

## **4.1 Verify Data Availability**

Before building any dashboards, confirm that alert data is flowing and present in Elasticsearch. On soc-stack (VM 600):

curl -sk -u elastic 'https://localhost:9200/_cat/indices/wazuh-alerts*?v&s=index'

You should see one index per day with a document count greater than zero. If indices are missing or empty, troubleshoot the Filebeat pipeline before proceeding.

Also confirm the total alert count across all indices:

curl -sk -u elastic 'https://localhost:9200/wazuh-alerts-4.x-*/_count?pretty'

## **4.2 Create the Baseline Data View**

In Kibana, navigate to Stack Management, then Data Views. Confirm that a data view covering wazuh-alerts-4.x-* exists with @timestamp as the time field. If it does not exist, create it now.

## **4.3 Build the Alert Volume Dashboard**

Navigate to Kibana Dashboards and create a new dashboard named SIEM Baseline. Add the following panels in order.

### **Panel 1: Total Alert Count (Metric)**

This gives you the headline number at a glance.

- Visualization type: Metric

- Data view: wazuh-alerts-4.x-*

- Metric: Count of records

- Time range: Last 7 days

- Title: Total Alerts (7 Days)

### **Panel 2: Alert Volume Over Time (Bar Chart)**

This shows you whether alert volume is stable, growing, or has spikes that correspond to specific events.

- Visualization type: Bar chart (vertical)

- X axis: @timestamp (auto interval)

- Y axis: Count of records

- Title: Alert Volume Over Time

### **Panel 3: Top 20 Rules by Alert Count (Table)**

This is the most important panel for baseline work. It gives you your ranked rule list.

- Visualization type: Table

- Rows: rule.id (top 20 values), rule.description (top 20 values)

- Metric: Count of records

- Sort: Count descending

- Title: Top 20 Rules by Alert Count

### **Panel 4: Alerts by Agent (Bar Chart)**

This tells you which hosts are generating the most noise, which helps you prioritize tuning per host in Phase 2.

- Visualization type: Bar chart (horizontal)

- Y axis: agent.name (top 10 values)

- X axis: Count of records

- Title: Alert Volume by Agent

### **Panel 5: Alerts by Severity Level (Donut Chart)**

This shows you the distribution of alert severity. A healthy baseline should be heavily weighted toward informational and low severity levels.

- Visualization type: Donut

- Slice by: rule.level (top 10 values)

- Metric: Count of records

- Title: Alert Distribution by Severity Level

### **Panel 6: Alerts by Rule Group (Bar Chart)**

Rule groups tell you what categories of activity are generating the most alerts. Authentication failures, file integrity events, and rootcheck findings are common high-volume groups.

- Visualization type: Bar chart (horizontal)

- Y axis: rule.groups (top 15 values)

- X axis: Count of records

- Title: Alert Volume by Rule Group

Save the dashboard.

## **4.4 Produce the Baseline Document**

Export the data from Panel 3 (Top 20 Rules) to a spreadsheet or document. For each rule in the list, record the following:

| **Rule ID** | **Description** | **Alert Count** | **Level** | **Initial Assessment** |
| --- | --- | --- | --- | --- |
| (from Kibana) | (from Kibana) | (from Kibana) | (from Kibana) | Expected / Investigate / Suppress |

The Initial Assessment column should be filled in based on your knowledge of your environment. An SSH login from your management workstation every morning is expected. A root login from an unknown IP is not. Do not suppress anything yet. That happens in Phase 2. This column is just your first reaction.

## **4.5 Identify High Priority Rules**

From your ranked list, flag any rule that meets any of the following criteria for immediate review before Phase 2:

- Severity level 12 or above, regardless of volume

- Any rule containing the words rootkit, malware, or injection in the description

- Any authentication failure rule firing more than 100 times per day on a single agent

- Any file integrity monitoring rule firing on a system binary path such as /bin, /sbin, or /usr/bin

| **WARNING** | If you find level 12 or above alerts in your baseline, do not suppress them during Phase 2 without first investigating their source. High severity alerts are high severity for a reason. Tune the threshold only if you have confirmed the activity is benign and understand exactly why the rule is firing. |

# **6. Troubleshooting**

## **Dashboard panels show no data**

Verify the data view time range. Kibana defaults to the last 15 minutes. Expand to Last 7 days. Also confirm the data view index pattern matches wazuh-alerts-4.x-* exactly including the wildcard.

## **rule.description field not available in Lens**

This field may not be mapped as a keyword in your index template. Check with:

curl -sk -u elastic 'https://localhost:9200/wazuh-alerts-4.x-*/_mapping/field/rule.description?pretty'

If the type is text rather than keyword, add rule.description.keyword as the field in Lens instead.

## **Alert counts seem unexpectedly low**

Check whether all agents are reporting. On wazuh-manager:

/var/ossec/bin/agent_control -la

Any agent showing as disconnected will not be contributing alerts. Investigate the agent service on that host.
